# `@fillament/i18n`

Localized labels, placeholders, and validation messages without coupling Fillament to a specific translation framework.

```bash
pnpm add @fillament/i18n
```

## Plain string support

Plain strings pass through untouched:

```ts
i18n.t("Email"); // → "Email"
```

## Message object support

Resolve a key from a dictionary; falls back to `fallback` (or the key itself) if the message is missing.

```ts
import { createI18n } from "@fillament/i18n";

const i18n = createI18n({
  locale: "pt",
  messages: {
    "user.email.label": "Email",
    "user.email.required": "O email é obrigatório",
  },
});

i18n.t({ key: "user.email.label", fallback: "Email" }); // → "Email"
```

## Interpolation

```ts
i18n.t({
  key: "field.minLength",
  values: { min: 8 },
  fallback: "Must be at least {min} characters",
});
// → "Must be at least 8 characters"
```

Tokens are `{name}` placeholders. Replace with a custom `interpolate` when you need a different syntax.

## Locale switching

```ts
const i18n = createI18n({
  locale: "en",
  messages: {
    en: { greet: "Hello {name}" },
    pt: { greet: "Olá {name}" },
  },
});
const unsub = i18n.subscribe(() => rerender());
i18n.setLocale("pt");
```

`createI18n` is observable — subscribe to be notified when the locale or dictionary changes.

## Fallback locale

```ts
createI18n({ locale: "pt-BR", fallbackLocale: "pt", messages: { ... } });
```

Missing keys in the active locale fall through to `fallbackLocale` before using the inline `fallback`.

## Custom resolvers

If you already have `intl`, `i18next`, or `lingui` configured, pass a resolver and skip the dictionary entirely:

```ts
createI18n({
  locale: "en",
  resolver: (key, locale) => intl.formatMessage({ id: key }) ?? undefined,
});
```

## One-shot helper

For dictionaries that never change at runtime:

```ts
import { resolveMessage } from "@fillament/i18n";

resolveMessage({ key: "x", fallback: "X" }, { messages });
```

## Optional adapters

`createIntlAdapter`, `createI18nextAdapter`, `createLinguiAdapter` are not bundled — each would pull a peer dependency. Implement them in your project against the same `I18nAdapter` shape if you need framework parity.
